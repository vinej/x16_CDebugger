This release adds **MADS (Mad Assembler)** as a ninth toolchain, with the same one-**F5** source-level debugging the other assemblers get: set breakpoints and drive execution with **F5 / F10 / F11 / Shift+F11** against real machine code in the Box16 emulator, via the [VS64](https://marketplace.visualstudio.com/items?itemName=rosc.vs64) extension.

Every toolchain still builds the same **bounce demo** (a sprite bouncing around the screen with PSG and YM2151 sound), and the MADS build is **byte-for-byte identical** to the ACME / ca65 / KickAssembler / dasm builds (MD5 `7F2F685178FD49D8C4F226BF2D09D3FD`).

## New in 0.2.0

- **MADS (Mad Assembler)** — full F5 source-level debugging, even though VS64 ships no MADS toolkit. MADS has no linker and emits a flat image, so the build prepends the `$0801` CBM load address; then `tools/mads2dbg.py` synthesizes a cc65-format `.dbg` from MADS's listing (`-l`) + label dump (`-t`), which VS64 loads by file extension. This is the same trick the `dasm/` project uses, adapted to MADS's listing format (basename `Source:`/`Macro:` markers, tab-delimited byte column, `+`-truncated data rows).
- The MADS bounce is a port of the shared demo from [x16_library](https://github.com/vinej/x16_library)'s MADS dialect (tag `v0.2.0`), generated by that repo's `tools/acme2mads.py` and validated against its 132-test on-target runner.

## Toolchains (nine)

**C toolchains** (build against the [x16clib](https://github.com/vinej/x16_clib) C library):
- **Oscar64** — full F5 debugging (globals/statics)
- **cc65** — full F5 debugging; the one toolchain whose debugger also shows **local variables**
- **llvm-mos** — full F5 debugging (globals/statics)

**Assembly toolchains** (build against the [x16lib](https://github.com/vinej/x16_library) assembly library):
- **ACME** — x16lib's reference dialect
- **ca65**
- **KickAssembler** (needs Java)
- **dasm** — full F5 debugging via a synthesized cc65-format `.dbg`
- **MADS** — *(new)* full F5 debugging via a synthesized cc65-format `.dbg`

**Prog8** — build/run plus *symbolic* debugging in Box16 (no F5 source-level debugging: VS64 has no prog8/64tass toolkit).

## How it works

Source-level debugging is enabled by a **Box16 fork** (branch `binary-monitor`) that adds a **VICE binary monitor server**, which VS64 talks to over TCP. This fork also carries monitor latency fixes (sub-millisecond paused servicing, `TCP_NODELAY`) and checkpoint extensions.

## Known limitations

- **Locals show only for cc65.** Oscar64 (`.dbj` carries no locals) and llvm-mos (VS64 doesn't extract DWARF locals) show globals/statics only.
- **Prog8** has symbolic debugging only, not F5 source-level stepping.
- **Stepping into library calls** is limited by VS64's cc65 debug-info resolver, which discards line records for sources not listed in `project-config.json`'s `sources`.

## Setup

The distribution is **source/config only**. Third-party binaries (compilers, emulators, the Box16 fork, the support libraries, ROM) are **not tracked in this repo** — after cloning, set them up once as described in the README's Setup section. MADS's `mads.exe` is not redistributable; drop it into `mads-sdk/` yourself (see the README). Nothing is installed system-wide except the VS64 extension itself.
